# 岩土工程 AI 辅助写作系统 — 技术设计文档

**版本：v3.0** | **日期：2026-03-30**

---

## 1. 系统概述

### 1.1 定位

本系统是一套面向岩土工程勘察报告场景的 AI 辅助写作系统。核心思路是**"规则提取 + 模板生成 + 工作流编排"**，而非开放式文本生成。

系统的核心差异化设计在于**参数锁定表机制**——它把"原始提取"和"最终写作"之间隔了一层可审查中间态，让 AI 写作建立在可核查的参数之上，而不是自由生成。

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| 可核查性 | 每一个输出数值都有来源，参数锁定表提供人工校核入口 |
| 安全降级 | 提取数据缺失时显示"待确认"占位，绝不编造数值 |
| 标准化输出 | 模板引擎保证措辞和格式符合岩土行业规范 |
| 可扩展性 | 模块化设计，新模板/新章节/新规则可独立添加 |
| 一键运行 | 从 PDF 到 Word 全链路自动化 |

## 2. 总体架构

系统采用六层架构，从下到上逐层组装：

```
┌──────────────────────────────────────────────────┐
│              ⑥ 成果交付层                         │
│         geo_word_exporter.py                      │
│    Markdown → Word，支持企业模板                   │
├──────────────────────────────────────────────────┤
│              ⑤ 业务工作流层                        │
│         geo_report_workflow.py                     │
│    PDF→提取→参数卡→章节→导出 全流程编排             │
├──────────────────────────────────────────────────┤
│              ④ 文档读取层                          │
│         opendataloader_pdf_reader.py               │
│    OpenDataLoader + pdfplumber 双引擎              │
├──────────────────────────────────────────────────┤
│         ③ 结论生成层                               │
│    geo_writer_conclusions.py (105条)               │
│    geo_writer_specialized.py (25函数)              │
├──────────────────────────────────────────────────┤
│         ② 标准表达层                               │
```

## 3. 模块设计

### 3.1 `geo_report_workflow.py` — 主工作流

系统入口和流程编排器。包含以下核心类：

**`PDFScanner`** — 扫描文件夹中的 PDF 文件，支持递归搜索和去重。

**`ProjectDataExtractor`** — 数据提取引擎，将 PDF 文本按 6 个模块分类抽取：
- A_项目基础信息：项目名称、地点、建筑规模、结构形式、勘察等级
- B_场地地层信息：层号、层名、厚度、埋深（当前较弱）
- C_原位测试信息：标贯击数、静探曲线（当前较弱）
- D_试验与参数：承载力特征值、压缩模量、桩基参数（成熟）
- E_地下水与腐蚀性：水位、抗浮水位、腐蚀性等级、离子含量（成熟）
- F_抗震与专项评价：烈度、加速度、分组、场地类别、波速、液化（成熟）

**`ProjectParameterCard`** — 参数锁定表生成器，将提取数据格式化为可审查的 Markdown 文档。提供 `_get()` 安全取值方法。

**`ReportChapterGenerator`**（v3.0）— 动态章节生成器，基于数据访问层从 `extracted_data` 驱动内容生成。详见第 7 节。

### 3.2 `opendataloader_pdf_reader.py` — PDF 读取引擎

双引擎设计：

```
优先：OpenDataLoader（Java，高精度表格提取）
降级：pdfplumber（纯 Python，无外部依赖）
```

核心类：
- `OpenDataLoaderReader`：调用 opendataloader-pdf，返回结构化文本和表格
- `PdfplumberReader`：pdfplumber 备选方案
- `get_best_reader()`：自动选择最优引擎

### 3.3 `geo_writer_templates.py` — 标准表达库

150 条参数化表达模板（EXP-001~150），按场景分类：

| 分类 | 编号范围 | 说明 |
|------|----------|------|
| 基础描述 | EXP-001~020 | 项目概况、地层描述 |
| 承载力 | EXP-021~050 | fak 建议、桩基参数 |
| 地下水 | EXP-051~070 | 水位描述、腐蚀性 |
| 抗震 | EXP-071~090 | 场地类别、液化判别 |
| 逐栋分析 | EXP-091~110 | G 类句式 |
| 专项评估 | EXP-111~150 | 施工风险、岩溶、管线、勘察局限性 |

每个模板函数采用 f-string 参数化设计，调用时传入实际数值。

### 3.4 `geo_writer_conclusions.py` — 结论模板库

105 条结论建议模板（CON-001~105），提供可直接使用的专业结论句式。支持建筑类编号格式和管线类圆圈格式两种风格。

### 3.5 `geo_writer_specialized.py` — 专用场景函数

25 个高频写作场景的专用函数，封装了完整的业务逻辑：

| 函数 | 用途 |
|------|------|
| `承载力建议()` | 地基承载力建议 |
| `桩基持力层建议()` | 桩基持力层选择（含首选/备选） |
| `腐蚀性结论()` | 腐蚀性综合评价 |
| `场地类别结论()` | 建筑场地类别判定 |
| `液化判别结论()` | 液化判别综合评价 |
| `场地稳定性结论()` | 场地稳定性评价 |
| `勘察等级结论()` | 岩土工程勘察等级判定 |

这些函数与章节生成器深度集成——章节生成器在组装文本时直接调用，将提取数据转化为专业表述。

### 3.6 `geo_word_exporter.py` — Word 导出器

Markdown → Word 转换器，功能包括：
- 标题层级自动映射（## → Heading 1，### → Heading 2）
- Markdown 表格 → Word 表格
- 基于企业模板创建文档（可选）
- 岩土报告标准字体和格式设置

### 3.7 `geo_logger.py` — 日志系统

统一日志接口，支持文件日志 + 控制台输出，包含步骤追踪、成功/失败标记、调试信息等。

## 4. 数据流设计

### 4.1 完整处理链路

```
① PDF 文件
   ↓ scan_pdfs() — 扫描文件夹
② 文本 + 表格
   ↓ extract_geotech_data() — 双引擎提取
③ 原始文本
   ↓ extract_module_A~F() — 正则抽取
④ extracted_data（A~F 六个模块字典）
   ↓ generate_parameter_card() — 格式化输出
⑤ 参数锁定表（Markdown）
   ↓ ReportChapterGenerator.__init__() — 数据访问层封装
⑥ 四个章节（Markdown）
   ↓ export_markdown_to_word() — 格式转换
⑦ 完整报告（Word）
```

### 4.2 数据模块映射

| 模块 | 数据来源 | 提取策略 | 成熟度 |
|------|----------|----------|--------|
| A_项目基础信息 | 报告封面、概况章节 | 关键词正则 | ⚠️ 需增强 |
| B_场地地层信息 | 地层统计表 | 表格解析 | ⚠️ 需增强 |
| C_原位测试信息 | 标贯/静探数据表 | 表格解析 | ⚠️ 需增强 |
| D_试验与参数 | 物理力学统计表 | 正则 + 表格 | ✅ 成熟 |
| E_地下水与腐蚀性 | 水位记录 + 水质分析 | 正则 + 多模式匹配 | ✅ 成熟 |
| F_抗震与专项评价 | 抗震评价章节 | 正则（13 个字段） | ✅ 成熟 |

## 5. 关键实现机制

### 5.1 文件名驱动的模块归类

`ProjectDataExtractor` 通过 PDF 文件名中的关键词（如"报告""勘察""正文"）识别主报告，排除附件和重复文件。

### 5.2 D/E/F 模块正则提取策略

- **D 模块**：匹配 `fak\s*=\s*(\d+)` 等承载力模式，关联层号
- **E 模块**：分干湿交替和长期浸水两种条件匹配腐蚀性，支持离子含量提取
- **F 模块**：13 个独立正则分别提取烈度、加速度、分组、场地类别、波速、液化等字段

### 5.3 参数锁定表机制

参数锁定表是系统的核心设计亮点：

1. **可审查性**：提取结果以 Markdown 表格形式显式展示，便于人工逐字段核对
2. **中间态隔离**：把"原始提取"和"最终写作"解耦，修改锁定表不影响提取逻辑
3. **质量兜底**：AI 写作基于锁定表参数，而非直接基于原始提取结果
4. **缺失透明**：未提取到的字段标记为"待确认"，明确暴露数据缺口

### 5.4 模板引擎三层结构

```
Layer 1: 150 条标准表达（geo_writer_templates.py）— 基础句式构件
Layer 2: 105 条结论模板（geo_writer_conclusions.py）— 专业结论句式
Layer 3: 25 个专用函数（geo_writer_specialized.py）— 封装完整业务逻辑
```

章节生成器优先调用 Layer 3 的专用函数（如 `桩基持力层建议()`），因为它们封装了更完整的业务上下文。

## 6. 动态章节生成机制（v3.0）

### 6.1 数据访问层

`ReportChapterGenerator` 通过统一的数据访问层从 `extracted_data` 取值：

```python
_safe_get(module, key, default="待确认")  # 通用安全取值
_get_A()  # 项目基础信息 — 13 个字段
_get_D()  # 试验参数 — 承载力表 + 桩基参数表 + 智能持力层选择
_get_E()  # 地下水与腐蚀性 — 水位 + 抗浮 + 腐蚀性 + 离子含量
_get_F()  # 抗震参数 — 烈度/加速度/分组/场地类别/波速/液化/冻结深度
```

### 6.2 三层取值逻辑

章节生成器对每个字段采用三层取值策略：

| 优先级 | 来源 | 行为 |
|--------|------|------|
| 1 | `extracted_data` 自动提取值 | 直接使用 |
| 2 | 参数卡默认值或合理默认 | 自动填充 |
| 3 | "待确认"占位 | 明确暴露缺失 |

这一设计确保：D/E/F 模块成熟时输出完整数据，A 模块尚弱时也不会输出编造的假数值。

### 6.3 四章节动态化范围

**第一章 工程概况** — 读取 A 模块 13 个字段：
项目名称、建设地点、工程性质、建筑规模、层数、结构形式、基础形式、基础埋深、基底压力、勘察等级、重要性/场地/地基等级。

**第六章 场地地震效应** — 读取 F 模块 13 个字段：
抗震设防烈度、地震加速度、设计地震分组、建筑场地类别、场地土类型、等效剪切波速、覆盖层厚度、特征周期、液化判别结论/层位、抗震地段类别、设防类别、标准冻结深度。

**第七章 地基基础分析** — 读取 D 模块动态生成：
- 承载力特征值表（按层号自动排列，含 Es0.1-0.2）
- 桩基设计参数表（qsik / qpk）
- 智能持力层选择（按 qpk 最大值自动推荐，含备选方案）

**第八章 结论与建议** — 综合 D/E/F 模块：
- 8.1 场地稳定性（调用模板函数）
- 8.2 地下水与腐蚀性（E 模块数据 + 专用函数）
- 8.3 地基方案建议（D 模块条件分支：有桩基数据→推荐方案 / 仅承载力→方案比选 / 无数据→提示补填）
- 8.4 施工建议（含液化条件提示）

### 6.4 智能决策示例

第七章的持力层选择不是简单模板填空，而是基于数据的规则决策：

```python
# _get_D() 内部逻辑：
# 1. 遍历所有层的桩基参数
# 2. 按 qpk（桩端极限阻力）降序排列
# 3. 选择 qpk 最大的层作为首选持力层
# 4. 选择 fak（承载力特征值）最大的层作为备选
```

这种设计将岩土工程师的经验判断固化为可执行规则。

## 7. 输出与交付机制

### 7.1 输出文件

| 文件 | 生成时机 | 格式 |
|------|----------|------|
| `{项目名}_项目参数锁定表.md` | 提取完成后 | Markdown |
| `第一章_工程概况.md` | 章节生成 | Markdown |
| `第六章_场地地震效应.md` | 章节生成 | Markdown |
| `第七章_地基基础分析.md` | 章节生成 | Markdown |
| `第八章_结论与建议.md` | 章节生成 | Markdown |
| `{项目名}_完整报告.docx` | 导出阶段 | Word |

### 7.2 Word 导出流程

1. 读取所有 Markdown 章节文件
2. 解析 Markdown 语法（标题、段落、表格、粗体）
3. 按章节顺序合并
4. 应用岩土报告标准样式
5. 输出为 Word 文档

## 8. 已知边界与扩展方向

### 8.1 当前边界

| 边界 | 说明 | 影响 |
|------|------|------|
| A/B/C 模块较弱 | 项目基础信息、地层信息、原位测试提取能力有限 | 第一章部分字段显示默认值 |
| 表格提取依赖 PDF 格式 | 非标准表格、跨页表格可能丢失 | 参数卡部分数据需人工补填 |
| 模板覆盖范围 | 当前 150+105 条模板覆盖主要场景 | 特殊岩土条件仍需人工补充 |
| 章节覆盖 | 当前生成 4 个章节（1/6/7/8） | 其余章节需后续扩展 |

### 8.2 扩展方向

| 方向 | 具体内容 | 价值 |
|------|----------|------|
| A/B/C 模块增强 | 结构化抽取 + 表格解析优化 | 提升全自动化率 |
| 章节扩展 | 覆盖 2~5 章和 9~10 章 | 实现完整报告生成 |
| 参数校验 | 添加合理性检查（如 fak 范围、水位逻辑） | 提升输出质量 |
| 模板分类细化 | 按建筑类型/地区/规范版本分类 | 提升模板精准度 |
| 企业模板适配 | 深度对接公司 Word 模板样式 | 减少排版工作 |
| 多项目批量处理 | 支持一次处理多个项目 | 提升工作效率 |
